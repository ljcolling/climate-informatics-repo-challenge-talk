What’s in a Badge?

An audit of one issue of Psych Science




Lincoln Colling

Introduction

  • Data sharing has become more common over the past few years

  • Possibly spurred on by the introduction of Open Data badges (e.g., see Kidwell et al., 2016)

Requirements for an Open Data Badge

The requirements for an Open Data Badge vary from journal to journal

But the minimal requirements are:

  1. Digitally shareable data on open-access repo (e.g., OSF).

  2. A data dictionary (for example, a codebook or metadata describing the data) in sufficient detail for an independent researcher to reproduce the reported analyses and results.

  3. An open licence allowing others to copy, distribute, and make use of the data while allowing the licensor to retain credit and copyright as applicable.

Requirements for an Open Data Badge

However, some journals are also have additional requirements


The Psychological Science submission guidelines specifically state that the open data badge awarded for:

for making publicly available the digitally shareable data necessary to reproduce the reported result. This includes annotated copies of the code or syntax used for all exploratory and principal analyses




… more on this later…

Do badges work as intended?

If the badges work as intended, then it should be possible to “reproduce the reported result” from the shared data…

This is what we set out to test…

In 2019, Psychological Science published its first issue where all 14 articles received an Open Data Badge

So, we decided to see if we could reproduce all the results


And the outcome was:

And so it began

It all started with a tweet from Nick Brown (@sTeamTraen) about a secret project…

  • Initially, 13 reproducers (including Nick) joined the project, with each indicating their expertise in Excel, Matlab, R, and SPSS

  • After two dropouts and one replacement, 12 reproducers were each assigned three to five papers depending on their expertise

  • Each paper had at least three reproducers, resulting in 46 individual reproduction reports

I was assigned four (3 behavioural or meta-analyses done in R and one EEG study done in Matlab)

The reproduction process

The process was split into two stages:

  1. Each reproducer independently attempted a reproduction of their assigned studies and wrote a report of their experience and findings

These reports all differed from reproducer to reproducer. Some reproducers included code, while others were mostly narrative

  1. Based on the individual reports, the group assigned to each article got together to write a summary report of their findings

Following this, each person rated the reproducibility of their assigned articles based on their individual experience and the group experience

Initial assessment

  • All the repositories linked to the articles appeared to contain at least some data

  • But 6 / 14 did not provide any code

We were off to a bad start…

Right off the bat, these 6 should not have been eligible for an open data badge, because they did not include “annotated copies of the code or syntax used for all exploratory and principal analyses”

… but more on this later…

The reproduction reports

All the reproduction reports are available on the OSF (osf.io/xzke7/)

The stage one reports were unstructured:

  • Some were more narrative in style

  • Others attempted to make reproducible documents of the articles results1

I won’t go into all the reports and articles in detail, but I’ll just pick out a few to discuss in more detail.

1I took the second route creating each report as an R Markdown document that included all the code to download the data from the OSF, perform the analyses, and produce the manuscript. This was all run automatically on GitHub using GitHub actions and Docker.

Article 110

EEG study examining working memory and attention

  • Shared data from 4 experiments

    • Shared data included EEG data and behavioural data

  • Shared experiment logs includes % of trials rejected and notes on EEG data quality

  • Shared MATLAB code for EEG behavioural data analysis, artifact rejection pipeline, and ERP analysis

Article 110 Summary

  • All data hosted in .zip files (a single failure point)

  • Analysis code contained hard-coded path names that needed to be edited

  • Some dependencies weren’t specified

  • Smoe of the reproduced t-tests differed in df (possibly because the data-file included summary statistics that were included in the published analysis)

  • Lack of clear meta-data linking datasets to the variables used in the analyses made it difficult to reproduce some analyses where code was missing

The recommendations were as follows:

Authors should provide code or syntax to exactly reproduce the statistical results reported in the paper from the data provided. For EEG data and other very large data files, the OSF is perhaps not the best repository - a neuro-specific open database such as OpenNeuro might be better. Finally and very importantly, authors should include a data and code check by an independent person to ensure that the results can be successfully reproduced from the contents of a repository.

The take home message from this is that sharing data in a way that makes reproduction easy is actually very difficult, and it required a certain amount of technical skill that many people are lacking. It didn’t seem like the data was shared with reproducibility in mind. You get the impression that author took the code and data that had worked on their machine. Zipped it all up. And dumped it on the OSF.

Article 113

Behavioural experiment using a dictator-game

  • Shared data includes a .csv file

  • No analysis code / SPSS syntax was provided.

  • Analysis included a power analysis, descriptive statistics, PCA, Wilcoxon tests, chi-squared tests, tobit regressions, and logistic regressions

Artlce 113 Summary

  • Although no code was provided the reproducer re-coded all the analyse in R

  • Most of the analyses were reproducible, except for logistic regression

  • The logistic regression was not reproducible because there wasn’t enough detail in the article to know which variables to include

  • One reproducer also failed to reproduce the power analysis for the Wilcoxin test because this wasn’t available in R (the other reproduced use G*Power)

  • No information on the software used was provided

Please cite the open source software you use, because RSEs don’t get enough credit for their work!

Article 114

A complex study including an aggression-testing paradigm, DNA tests, administration of testosterone or placebo, and psychometric testing.

  • Shared data includes 3 .xlsx files

  • Analysis code included but cut-and-pasted into the .xlsx files

  • Analysis included a ANOVAs and t-tests

Article 114 Summary

  • This article was fairly easy to reproduce using the supplied R code

  • And it was easy to re-code in R from the description in the article… which is what I did because I didn’t know they had included code in the .xlsx until after re-coding it all!

  • Only heavily pre-processed data was included, and providing raw data, and code for pre-processing, would’ve been an improvement

Code belongs in code files and data belongs in data files! Reusing code and data shouldn’t involve a manual cut and paste job!

Assessing the effectiveness of badges

  • The aim of our study was to measure the effectiveness of the Open Data Badge policy

  • Our reproducibility ratings were a proxy for effectiveness

  • We didn’t have formal criteria but used the following as a guide:

results should at least be reproducible by a competent external researcher such as a PhD Student

Summary of results

  • 1 of 14 was rated as exactly reproducible1

  • An additional 3 were rated as essentially reproducible with minor deviations

  • An additional 6 were rated as partially reproducible

    • Some of the partially reproducible are the most interesting because they highlight some of the issues of sharing data and code

    • Some reproducers had more experience with some analyses than others and were better able to fill in the blanks

  • Of the remaining, 1 was rated mostly not and 3 were rated not at all

  • Exactly reproducible was the ideal. No deviations from the reported results.

  • Essentially reproducible meant that there were minor deviations in the decimals or obvious typographical errors

  • Partially reproducible meant that there were more than minor deviations but that the results were mostly numerically consistent.

  • Mostly not reproducible meant that there were major deviations and few numerically consistent results.

  • Not at all reproducible meant that there were no numerically consistent results or the reproduction attempt was otherwise not possible.

  1. Note that will this paper received a rating of exactly reproducible it wasn’t completely without issues!

Summary of Issues

  • Lack of documentation of data and/or code

  • Where minor discrepancies occurred some of it could possibly be a result of not using fixed random seeds for e.g., bootstrapped analysis

  • Other minor errors might be copy-paste errors

  • Where code was not provided, analyses were often not reported in sufficient detail to reproduce (e.g., not reporting variables used, not reporting exclusion/inclusion criteria, using one-sided tests without stating it)1

  • Data storage issues on OSF

  1. In some cases, authors incorrectly described an analysis but correctly reported the analysis as specified in the code!

Data specific issues

  • Not providing raw data and only providing heavily pre-processed data

  • Providing raw data without providing cleaned data (when data cleaning procedures not specified)

  • Data not correctly labelled

Code specific issues

  • Lack of analysis and modelling code

  • Not including dependencies

  • Version of packages not specified1

  1. Even the paper that was rated as exactly reproducible was only reproducible when using (at the time) 2 year old versions of specific R packages

Jumping back to the badge requirements

Psychological Science submission guidelines state that articles may receive this badge

“for making publicly available the digitally-shareable data necessary to reproduce the reported result. This includes annotated copies of the code or syntax used for all exploratory and principal analyses.”

Psychological Science, 2022, Open Science Badges section; eligibility criteria in operation in 2019

But was there code?

  • Based on Psych Science’s own eligibility criteria for awarding an open data badge 8 didn’t meet the criteria

  • But it turns out that there was a mismatch between the author guidelines and the Open Practices Disclosure form (which served as the basis for awarding badges)

  • Neither the OSF guidelines not the OPD made references to code.

Some good outcomes

  • Our paper has lead to Psychological Science updating their OPD so that it provides better guidance

    • This is great! Because making your work reproducible is hard, and authors need proper guidance on it.

  • And several authors of papers in this issue have updated their repositories to address issues that we identified.

    • This is also great! Because scientists want to do the right thing, but you sometimes don’t know what you don’t know, and things that seem obvious to you might be completely opaque to outsiders1
1At Sussex Psychology we actually have a pilot programme were researchers can get independent reproducibility checks to help them spot these gaps.
